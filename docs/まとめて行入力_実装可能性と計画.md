# まとめて行入力画面 — 実装可能性の判断と実装計画

## 1. 結論：**実装可能**

技術的・既存アーキテクチャ上、別画面として要件を満たすことは可能です。新しいサーバーAPIは必須ではありません（下記参照）。

---

## 2. 認識合わせ（仕様確定分）

### 2.1 「確定」＝登録ボタンは **画面に1か所だけ**（各行には不要）

- **以前の案A/Bで言うと、「案B」に近い**：ドラフト行を複数保持し、**1つの「登録」操作**で `batch` へ送る。
- **「プラスを押すと確定ボタンが出る」の意味**  
  - ページ初期表示では **登録ボタンは出さない**（何も入力していないのに出ているとおかしい）。  
  - ユーザーが **下部の＋** または **コピー** によってドラフト行が生え、**編集モード（＝未登録行が存在する状態）** になったタイミングで、**画面の1箇所**（例：テーブル下）に **登録ボタンを表示**する。
- **データ構造**は `draftRows[]` の1段でよい（行ごとの「確定でステージへ」は不要）。

### 2.2 残高列（個人・施設）

- 未保存のドラフト行は **`—`**（登録後に明細再取得で実績が入る）。この方針で進める。

### 2.3 UI のゴール表現

- **フルスプレッドシート相当はスコープ外**。正解は **インライン入力**（行内にコントロールを置く）。  
- 目的：**1行作ってコピーし、あとは入力しやすくする**。

### 2.4 業務ルール・画面挙動は **現行 `まとめて入力（bulk-input）` と同様**

- 締め済み月の扱い、**訂正入力**（過去月側）の要否、**対象日の入力可能範囲**（10日ルール等）、取引種別の扱い、明細の **訂正／コピー**（登録済み行）の可否・条件などは **`app/facilities/[id]/bulk-input/page.tsx` と揃える**。
- **ヘッダーの「入金」「出金」ボタンと入出金モーダル**は、当初要件どおり **新画面では置かない**（区分はドラフト行のインラインで選択する想定）。それ以外の**制約・メッセージ・API**は現行に合わせる。

## 3. 既存資産との整合

| 要素 | 現状（まとめて入力） | 新画面での活用 |
|------|---------------------|----------------|
| 一括登録 | `POST /api/transactions/batch`（最大100件、`lib/transactionCreateValidation` と同一検証） | そのまま利用 |
| ストック | `PendingTransaction[]` + モーダル | **同型のデータ**を `draftRows` としてテーブル内インライン編集で保持し、1回の登録で送信 |
| 明細表示 | `/api/facilities/{id}/transactions?year=&month=` | 同じ fetch 構成で再利用 |
| 利用者・ユニット | `/api/residents`, `/api/units` | 行内の利用者選択（検索・ユニット絞り込み）に再利用 |
| コピー（登録済み→ドラフト） | 現行はフォーム向けに利用者・金額を空 | **新画面では全項目コピー**で1ドラフト行を生成（現行の「まとめて入力」上のコピー仕様とは別要件） |

## 4. 要件ごとの実現方針（更新）

1. **レイアウト**  
   - 新ルート（例: `/facilities/[id]/bulk-row-input`）。戻る・施設名・月表示・明細テーブルは `bulk-input` と揃える。  
   - ヘッダーの入金/出金ボタン・入出金モーダルは出さない。

2. **ドラフト行と＋**  
   - 登録済み明細の**最終行の下**に **＋** を配置。押下で空のドラフト行を1行追加。  
   - **コピー**も同様にドラフト行を1行追加（全項目コピー）。

3. **登録ボタン**  
   - `draftRows.length === 0` のときは **非表示**（または disabled で隠す）。  
   - **1件以上**になったら表示。クリックで **全ドラフト行をバリデーション**し、問題なければ `batch` で一括登録。

4. **登録済み行**  
   - 読み取り専用。現行と同様に **訂正・コピー**（条件付き）は維持。

5. **複数行**  
   - ドラフトは複数保持可能。登録は常に **1ボタンでまとめて**。
