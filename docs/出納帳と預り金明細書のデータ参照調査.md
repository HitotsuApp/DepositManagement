# 出納帳と預り金明細書のデータ参照調査

## 調査目的
「出納帳だけ描画順が逆になる」問題について、出納帳と預り金明細書で参照するデータの違いを調査する。

---

## 1. データフロー概要

### まとめて印刷 (batch-print) のレスポンス構造
```json
{
  "facilitySummary": { /* 出納帳用 */ },
  "residentStatements": [ /* 預り金明細書用、利用者ごとに1要素 */ ]
}
```

### 出納帳 (facilitySummary) の生成
- **API**: `GET /api/print/batch-print`
- **生成**: `transformToPrintData(sortedFacility, null, year, month)`
- **含まれるデータ**: transactions, summary, unitSummaries, grandTotal など

### 預り金明細書 (residentStatements) の生成
- **API**: 同上（同一レスポンス内）
- **生成**: `sortedResidents.map(r => transformToResidentPrintData(residentWithRelations, ...))`
- **含まれるデータ**: 各利用者ごとの transactions, summary など（unitSummaries は含まない）

---

## 2. ソート処理の比較

### batch-print API 内のソート（両方のデータソース）
```javascript
// 52-61行目
const sortedUnits = sortUnitsForPrint(facility.units, useSameOrder)
const sortedResidents = sortResidentsForPrint(
  facility.residents,
  facility.units,  // ← ソート前の Prisma の units（配列順は id 順の可能性）
  useSameOrder,
  useUnitOrder
)
const sortedFacility = { ...facility, units: sortedUnits, residents: sortedResidents }
```

### transformToPrintData 内のソート（出納帳のみ）
```javascript
// 81-90行目
const targetUnits = sortUnitsForPrint(facility.units, useSameOrder)
// facility.units = sortedUnits（API でソート済み）→ 同じ結果

const targetResidents = sortResidentsForPrint(
  facility.residents,
  facility.units,  // ← sortedUnits
  useSameOrder,
  useUnitOrder
)
// facility.residents = sortedResidents → 同じ結果
```

**結論**: ソートロジックは同一。sortResidentsForPrint の第2引数 `units` は unitMap の構築にのみ使用され、配列順序はソート結果に影響しない（displaySortOrder の値で比較）。

---

## 3. 参照データの違い

| 項目 | 出納帳 (facilitySummary) | 預り金明細書 (residentStatements) |
|------|--------------------------|-------------------------------------|
| **データソース** | transformToPrintData(sortedFacility) | transformToResidentPrintData(residentWithRelations) × N |
| **facility の units** | sortedUnits | 各 resident の unit のみ（配列ではない） |
| **facility の residents** | sortedResidents（全員） | 使用しない（1人ずつ処理） |
| **unitSummaries** | あり（targetUnits.map で生成） | なし |
| **ページ順序の決定** | 1ページ（複数ページの場合は transactions の chunk） | residentStatements の配列順 = sortedResidents の順 |

---

## 4. unitSummaries の生成フロー（出納帳のみ）

```
targetUnits (河瀬, 雲母, 筧)
    ↓ .map(unit => ...)
unitSummaries[0] = 河瀬の集計 (unitResidents = targetResidents.filter(r => r.unitId === 23))
unitSummaries[1] = 雲母の集計
unitSummaries[2] = 筧の集計
```

- `targetResidents.filter(r => r.unitId === unit.id)` は **filter の仕様で元配列の順序を保持**
- targetResidents は (unitOrder, residentOrder) でソート済み
- → 各ユニット内の residents は正しい順序

---

## 5. PDF レンダリングの違い

### 出納帳のレンダリング
```
BatchPdfRenderer
  → renderPages(depositStatementTemplate, data.facilitySummary, "facility")
  → data = facilitySummary（unitSummaries を含む）
  → 最終ページで UnitSummaryBlock unitSummaries={data.unitSummaries}
```

### 預り金明細書のレンダリング
```
BatchPdfRenderer
  → residentStatements.flatMap((residentData, index) =>
      renderPages(residentStatementTemplate, residentData, `resident-${index}`))
  → unitSummaries は使用しない（resident-statement テンプレートには UnitSummaryBlock なし）
  → ページ順 = residentStatements の配列順 = sortedResidents の順
```

**重要な違い**: 預り金明細書は「ページの並び」で順序が決まる。出納帳は「1つのページ内の UnitSummaryBlock」で unitSummaries の配列順が描画順になる。

---

## 6. 想定される原因

### A. UnitSummaryBlock の描画順
- `unitSummaries.map(unit => ...)` で配列順に描画
- 現在 `displayOrder = [...unitSummaries].reverse()` で反転を試行
- **仮説**: @react-pdf/renderer の View の子要素の描画順が、何らかの理由で逆になっている可能性

### B. データの受け渡し経路
- facilitySummary は API → fetch → setBatchPrintData → BatchPdfRenderer と渡る
- JSON のシリアライズ/デシリアライズで配列順が保たれるか要確認
- **通常は保たれる**が、クライアント側で加工している箇所がないか確認

### C. 出納帳の transactions テーブルの順序
- allTransactions は日付順でソート（同一日付は id 順）
- ユニット順・利用者順は取引テーブルには反映されない（時系列が優先）
- 「逆」が取引テーブルを指す場合は別原因

### D. 単体出納帳 vs まとめて印刷
- 単体: `/api/print/deposit-statement` → PdfRenderer
- まとめて: `/api/print/batch-print` → BatchPdfRenderer
- 両方とも renderPages 内で UnitSummaryBlock に data.unitSummaries を渡す
- データ構造は同じ想定

---

## 7. 推奨アクション

1. **UnitSummaryBlock の reverse を削除**して、生データの順序で描画した結果を確認
2. **ブラウザの開発者ツール**で、batch-print API のレスポンスの `facilitySummary.unitSummaries` の配列順を確認
3. **単体の出納帳印刷**（deposit-statement API）でも同様に逆になるか確認し、BatchPdfRenderer 固有かどうか切り分け
4. **@react-pdf/renderer のバージョン**と、flexDirection や描画順に関する既知の不具合を調査
