# 現金確認印刷：「本数（５０枚）」改行問題の調査

## 調査目的
現金確認画面から印刷した際、「本数（５０枚）」が改行されてしまう問題の根本原因を特定する。

---

## 1. 印刷の仕組み

### 現金確認画面の印刷方式
現金確認画面では **ブラウザの `window.print()`** により、**画面上のHTMLをそのまま印刷**しています。

- **印刷ボタン**: `handlePrint` → `window.print()`（`app/cash-verification/page.tsx` 167行目）
- **表示/非表示の制御**: `@media print` のCSSで、ヘッダー・サイドバー・ボタンなどを非表示にし、印刷対象のみ表示
- **PDF出力**: `/api/print/cash-verification` のAPIは存在するが、現金確認画面の印刷ボタンからは呼ばれていない

つまり、**ブラウザがレンダリングしたHTMLをそのまま印刷**しており、PDFレンダラー（`CashVerificationPdfRenderer`）は使われていません。

---

## 2. 問題の発生箇所

### 該当コード
`app/cash-verification/page.tsx` 474-478行目:

```tsx
<thead>
  <tr className="border-b">
    <th className="text-left py-2 px-4">【本】</th>
    <th className="text-center py-2 px-4 w-32">本数（50枚）</th>  <!-- ← ここ -->
    <th className="text-right py-2 px-4 w-40">金額</th>
  </tr>
</thead>
```

### 比較：紙幣テーブル（改行されない）
同じ `w-32` が指定されているが、「枚数」は2文字のため問題なし:

```tsx
<th className="text-center py-2 px-4 w-32">枚数</th>
```

---

## 3. 根本原因の分析

### 3.1 セル幅の制約
- **`w-32`** = 8rem = **128px**（1rem = 16px の場合）
- **`px-4`** = 左右それぞれ 1rem = 16px → 合計 32px のパディング
- **テキストが入れる実効幅** ≈ 128 - 32 = **96px**

### 3.2 テキストの必要幅
「本数（50枚）」= 7文字

| 文字種 | 文字数 | おおよその幅（1em = 16px） |
|--------|--------|---------------------------|
| 全角（本・数・（・枚・）） | 5文字 | 5 × 16 = 80px |
| 半角（5・0） | 2文字 | 2 × 8 = 16px |
| **合計** | 7文字 | **約96px** |

- 実効幅 96px に対して、必要幅が約96px → **ギリギリまたは不足**
- フォントやブラウザによって文字幅は変動するため、環境によって改行が発生

### 3.3 環境差が生じる理由

1. **フォントの違い**
   - プロジェクトでは `Hiragino Sans`, `Meiryo` などが指定されているが、OS・ブラウザによって実際に使われるフォントが異なる
   - 印刷時はブラウザやOSの印刷用フォントが使われる場合があり、文字幅が変わる

2. **印刷時のスケール・DPI**
   - 印刷プレビューや実際の印刷では、画面表示と異なる解像度・スケールが適用される
   - その結果、セルの実効幅が画面表示時より狭く計算されることがある

3. **テーブルレイアウト**
   - `table-layout: auto`（デフォルト）では、列幅は内容に応じて自動計算される
   - 親要素の幅制約（`w-full` + `overflow-x-auto`）や印刷時のページ幅により、列幅が圧縮される

4. **全角 vs 半角**
   - ソースコードでは「本数（50枚）」と半角の「50」を使用
   - ユーザー環境で全角「５０」として表示・印刷される場合（フォント置換など）、幅がさらに増え、改行が起こりやすくなる

---

## 4. 結論：根本原因

**根本原因は「セル幅（w-32 = 128px）に対して、テキスト「本数（50枚）」の必要幅がギリギリまたは超過していること」です。**

- 文字の表示範囲（セル幅）が狭く、テキストが収まりきらない
- デフォルトの `white-space: normal` により、収まらない部分は改行される
- フォント・ブラウザ・印刷設定の違いで、環境によって改行の有無が変わる

---

## 5. 解決策の比較

| 対策 | 効果 | 備考 |
|------|------|------|
| **半角カッコに統一** `本数(50枚)` | ○ 幅が約16px減り、収まりやすくなる | 根本解決にはならないが、改善の可能性あり |
| **セル幅を広げる** `w-32` → `w-40` (160px) | ◎ 余裕を持って収まる | レイアウト変更が必要 |
| **改行を禁止** `whitespace-nowrap` | △ はみ出す可能性、レイアウト崩れ | 推奨しない |
| **フォントサイズを小さく** | △ 可読性低下 | 推奨しない |
| **PDF印刷に切り替え** | ◎ PDFは固定レイアウトで安定 | 実装コストがかかる |

---

## 6. 推奨対応

1. **即効性のある対応**: セル幅を `w-32` から `w-40`（160px）に変更し、テキストが余裕を持って収まるようにする
2. **補助的な対応**: 半角カッコ `(50枚)` に統一することで、さらに余裕を持たせる（任意）
3. **中長期**: 印刷品質を重視する場合は、PDF出力への切り替えを検討

---

## 7. 参考：PDFレンダラーの場合

`pdf/renderer/CashVerificationPdfRenderer.tsx` では、列幅を **パーセンテージ（30%）** で指定しており、A4用紙幅に対して固定レイアウトで描画されます。PDFはブラウザのレンダリングに依存しないため、改行問題は発生しにくい構造です。

ただし、現金確認画面の印刷ボタンは `window.print()` を使用しているため、このPDFレンダラーは現状使われていません。
