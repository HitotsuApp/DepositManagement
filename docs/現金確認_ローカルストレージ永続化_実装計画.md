# 現金確認：枚数・本数の localStorage 永続化 — 可否と実装計画

## 結論：**対応可能**

ブラウザの **localStorage** に、施設 ID ごとに枚数・本数（実質は各金種の `count`）を保存すれば、要望どおり実現できます。

- ブラウザを閉じても、**リセットするまで**保持（同一端末・同一オリジン・ストレージが消えない限り）。
- **施設が変わるたび**、その施設用の保存領域を読み書きすれば、施設ごとに独立して保持できる。
- **月をまたいでもよい**なら、キーに年月を含めず `facilityId` のみでよい（現状の仕様と一致）。

---

## 現状（コード上の前提）

| 項目 | 内容 |
|------|------|
| 画面 | `app/cash-verification/page.tsx`（`'use client'`） |
| 入力状態 | `bills` / `coins` の `useState`（メモリのみ） |
| 施設 | `selectedFacilityId`（`FacilityContext` または画面内ローカル選択） |
| リセット | `resetCounts()` で金種を初期テンプレートに戻している |

画面遷移で保持されるのは、Next.js のクライアントコンポーネントがアンマウントされないナビゲーションのためと考えられる。タブを閉じると state は失われる。

---

## 設計方針

### 1. 保存キー

例（アプリ名で衝突を避ける）:

```text
deposit-mgmt:cash-verification:v1:facility:{facilityId}
```

- `v1` … 将来フォーマット変更時にマイグレーションしやすくする。
- `facilityId` … 施設ごとに別エントリ。

### 2. 保存する値

金額（`amount`）は `count` から再計算できるため、**`count` の配列だけ**保存すれば足りる。

- `billCounts: number[]`（長さ = 紙幣行数、順序は `BILL_DENOMINATIONS` と同一）
- `coinCounts: number[]`（長さ = 硬貨行数、順序は `COIN_DENOMINATIONS` と同一）

読み込み時は配列長・数値妥当性を検証し、不正なら破棄して初期値（0）にフォールバック。

### 3. いつ読み込むか

`selectedFacilityId` が変わったとき（および初回で ID が確定したとき）に、その施設キーから読み込み、`setBills` / `setCoins` でマージする。

### 4. いつ保存するか（重要：施設切り替え時の取り違え防止）

`useEffect([selectedFacilityId, bills, coins])` で一律保存すると、**施設を変えた直後の 1 フレームで「新しい施設 ID に古い施設の入力」を保存してしまう**リスクがある。

推奨は次のいずれか（実装は A が単純）。

| 方式 | 内容 |
|------|------|
| **A. 変更ハンドラ＋リセットのみで保存** | `handleBillCountChange` / `handleCoinCountChange` の末尾と `resetCounts` 内で `localStorage.setItem`。施設変更時は読み込みのみ。 |
| **B. フラグ付き effect** | ストレージからの復元直後は `isRestoringRef` で保存 effect をスキップ。 |

### 5. リセット時

- 画面上は現状どおり `count` / `amount` を 0 に戻す。
- **同じ施設キーの localStorage を削除するか、ゼロ配列を書き込む**（どちらでもよいが、削除ならキーが残らない）。

### 6. `selectedFacilityId === null` のとき

保存・読込は行わない（または空表示）。誤ってキー `facility:null` に書かないよう注意。

### 7. ブラウザ・運用上の限界（共有端末など）

- localStorage は **そのブラウザ・そのオリジン（ドメイン）** に紐づく。別端末・別ブラウザ・シークレットモード終了後は共有されない／消える場合がある。
- **共有 PC** では、金額に近い情報が端末に残る点を運用で許容するか、必要なら「ログアウト時にクリア」など別要件で検討。

### 8. セキュリティ

サーバに送らないローカル下書きであり、機微度はアプリ全体方針に依存。XSS があると localStorage は読まれうるため、**XSS 対策は従来どおり重要**（本件で増えるリスクというより、保存場所が増える点の認識共有）。

## 実装タスク一覧（想定）

1. **ユーティリティ**（例: `lib/cashVerificationStorage.ts`）  
   - キー生成、`load` / `save` / `clear`、バリデーション、JSON パース失敗時の扱い。

2. **`page.tsx` 改修**  
   - `selectedFacilityId` 確定・変更時に `load` → state 反映（`amount` 再計算は既存ロジックと同じ式）。  
   - 入力変更・リセット時に `save` / `clear`。  
   - `typeof window !== 'undefined'` で SSR では触れないようにする（本ファイルは client だが、将来の分割に備えて防御的に）。

3. **動作確認**  
   - 施設 A で入力 → 別画面へ → 戻る → 値が残る。  
   - ブラウザを閉じて再度開く → 残る。  
   - 施設 B に切り替え → B 用の値（未入力なら 0）。A に戻す → A の値が復元。  
   - リセット → 0 になり、再訪問でも 0。  
   - プライベートブラウジング・ストレージ無効時は例外を握りつぶして従来どおりメモリのみ（コンソール警告程度）も検討。

4. **（任意）** 仕様書・マニュアルに「端末ローカルに保存される」旨を 1 行追記。

## まとめ

| 要望 | 可否 |
|------|------|
| リセットまでブラウザを閉じても保持 | 可能（localStorage） |
| 施設ごとに保持 | 可能（キーに `facilityId`） |
| 月をまたいでもよい | 可能（キーに年月を含めない） |

実装規模は **小〜中**（専用モジュール 1 ファイル＋現金確認ページの読み書きフック）で収まる見込み。
